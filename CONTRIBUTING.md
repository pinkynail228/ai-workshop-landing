# Правила Работы С Репозиторием

Этот репозиторий деплоится в production на `https://ailending.vercel.app` через Vercel + GitHub.

## 1. Базовые правила

1. `main` всегда должен быть deploy-ready.
2. Любые изменения делаются только через отдельную ветку и Pull Request.
3. Прямые коммиты в `main` запрещены, кроме аварийного hotfix.
4. Один PR = одна логическая задача.

## 2. Структура проекта

1. В корне репозитория храним:
   - `index.html`
   - `styles.css`
   - `script.js`
   - `hero.png`
   - `grok-video-5ed9e880-8975-466b-a3cd-6b8d54e036e5.mp4`
2. Локальные служебные файлы не коммитим:
   - `.vercel/`
   - `.DS_Store`

## 3. Ветки и коммиты

1. Перед началом работы:
   - `git switch main`
   - `git pull --ff-only`
2. Создание ветки:
   - `git switch -c codex/<type>-<short-description>`
   - Примеры: `codex/feat-add-cta`, `codex/fix-mobile-spacing`
3. Формат коммитов:
   - `<type>: <short summary>`
   - Типы: `feat`, `fix`, `docs`, `chore`, `refactor`
   - Пример: `fix: align hero badges on mobile`

## 4. Pull Request процесс

1. После изменений:
   - `git add -A`
   - `git commit -m "<message>"`
   - `git push -u origin <branch>`
2. Создаем PR в `main`.
3. В PR обязательно:
   - короткое описание “что и зачем”;
   - список измененных файлов;
   - скриншоты до/после для визуальных правок;
   - чек-лист проверок (см. ниже).
4. Мержим только после self-review.

## 5. Обязательные проверки перед PR

1. Локальный запуск:
   - `python3 -m http.server 8123`
   - открыть `http://127.0.0.1:8123`
2. Проверить:
   - страница открывается без ошибок;
   - все ассеты грузятся;
   - адаптив (desktop + mobile);
   - нет битых ссылок и случайных удалений.
3. Убедиться, что в коммите нет лишних файлов.

## 6. Правила деплоя

1. Push в `main` = production deployment в Vercel.
2. Любые рискованные изменения сначала через PR.
3. Если выкладка сломала прод:
   - откат через Vercel (`rollback`/promote предыдущего деплоя);
   - отдельный `hotfix` PR.

## 7. Аварийный hotfix

1. Ветка: `codex/hotfix-<short-description>`.
2. Минимальный дифф только под инцидент.
3. После выкладки обязательно сделать cleanup PR с нормализацией изменений.

## 8. Безопасность

1. Не хранить токены, ключи и личные данные в репозитории.
2. Не коммитить файлы окружения и локальные конфиги.
